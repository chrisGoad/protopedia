JavaScript Prototypes in the Open

Prototypes play a central role in JavaScript, but are unseen except by coders. At least, end-users have had no systematic view of this realm via user interfaces. One reason: lack of uniform structure of the object graph within which the prototype chains appear.

The issue can be addressed by restricting attention to structures which are trees in their object/own-property/value aspect (but which, unlike JSON objects, may contain prototype chains and functions). Then, via user interface, end-users can be given new powers, namely access to all prototypes and instances underlying an application.

The chart, when viewed in the UI, manifests the complete adjustability that arises from the above-mentioned powers. Of course, all kinds of adjustability can be built into user interfaces via application-specific coding, but in the case of PrototypeJungle, the treatment is uniform.

PrototypeJungle provides a way of assembling such structures efficiently via its component system.

The components are objects which obey the same structural rule as the whole application: they must be tree-like in their object/own-property/value aspect. PrototypeJungle provides a serialization method, so the components can be stored. Components may themselves serve as prototypes, since PrototypeJungle extends instantiation, normally a one-level operation implemented by Object.create or new, to hierarchical structures. The scheme results in highly adjustable applications, as has been seen in the example.

Here is a diagram illustrating the structure of a component or application.

Precisely stated: a red arrow runs from node a to node b if Object.getPrototypeOf(a) === b. A black arrow runs from a to b if a.P === b for some own property P of a. The requirement is that the black and green subgraph (whose edges consist of object/own-property/value relationships), be a tree. Such structures are dubbed, loosely but concisely, "prototype trees".
